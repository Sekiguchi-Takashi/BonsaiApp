# v1.0 — フェーズ3（RAG応答パイプライン）

これで当初の構想が一周した。

```
Gmail(IMAP IDLE) → キュー → BM25検索 → プロンプト組立 → Bonsai推論 → アプリ内に表示
```

## 追加: `mail/Pipeline.kt`

未処理メールを1件ずつ処理する。

### 文脈の予算管理（ここが要）
`n_ctx` が 2048 しかないので、資料を詰め込みすぎると質問本文が
押し出されて回答が破綻する。そのため上限を明示的に切っている。

| 用途 | 上限 |
|---|---|
| 参考資料 | 1100文字 |
| メール本文（質問） | 600文字 |
| 回答生成 | 400トークン |
| 検索で拾うチャンク | 上位3件 |

上位から順に、予算が尽きるまで詰める方式。

### 検索クエリの作り方
件名を2回繰り返して重み付けし、本文は**先頭300文字**だけ使う。
メールは冒頭に用件が来て、末尾は署名で埋まることが多いため。

### プロンプト
システム指示で以下を明示している。
- 日本語のみで書く
- 参考資料を根拠にする
- **資料に無ければ推測せず「該当する記載がありません」と書く**
- 要点を先に書く

温度は 0.3（通常の生成より低い）。事実ベースの回答なので発散させない。

回答末尾に参照したファイル名を付ける。

## サービス側

- メール監視ONで**推論スレッドも一緒に起動**する。
  受信だけしても意味がないため
- モデル未読込なら5秒待って再試行（サービス自前の読込完了を待つ）
- 常駐通知に「受信 / 推論」の両方の状態を表示
- 回答完了時に**別枠の通知**を出す（タップでメール画面へ）
- メール監視OFFで推論スレッドも停止

## UI

- キュー一覧に**回答本文を表示**（✓ 完了 / … 未処理 / × 失敗）
- 「失敗した項目を再処理」ボタン
- ステータスに受信状態・推論状態・件数を表示

## 動作確認の順番

1. RAG設定でインデックスを作る（文書数が0だと資料なしの回答になる）
2. モデルを読み込む
3. メール連携 → 監視を開始
4. 自分宛にテストメールを送る
5. 数秒でキューに現れ、「処理中」→「✓」と進む

## 想定される所要時間

1通あたり **検索は数十ms**、**推論が支配的**。
1.7B で20〜40秒、4B で1〜2分、8B で3〜5分が目安。
処理は直列なので、連続受信すると滞留する。

## 既知の制約

- 回答の自動返信はしない（アプリ内表示のみ。仕様どおり）
- 添付ファイルは扱わない
- スレッド（過去のやり取り）は考慮しない。1通ごとに独立して処理する
- 資料が長文の場合、BM25の上位3チャンクに答えが入らないと外す。
  ここが実運用での主な精度上の課題になる
