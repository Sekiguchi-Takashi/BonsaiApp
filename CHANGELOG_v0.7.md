# v0.7 — RAG フェーズ1（OneDrive差分同期 + BM25）

3フェーズ計画の1段目。LINE連携とLLM接続はまだ入っていない。
**この段階だけで検証できる**ようにしてある。

## 入ったもの

### OneDrive 差分同期（`rag/OneDrive.kt`）
- **device code flow** で認証。MSAL 不使用、素の HTTP のみ。
  リダイレクトURI不要なのでモバイル単体で完結する
- Graph の **delta API** で差分取得。`deltaLink` を保存し、
  2回目以降は変更分だけ取る
- `.txt` / `.md` / `.markdown` / `.text` のみ対象、2MB上限
- 削除は `deleted` フラグを見てインデックスからも除去
- トークンは自動リフレッシュ

### BM25 インデックス（`rag/RagDb.kt` / `rag/Tokenizer.kt`）
- **埋め込みモデル不要**。追加RAM消費はほぼゼロ（数MB）
- SQLite の転置インデックス + BM25 スコアリング（k1=1.2, b=0.75）を自前実装
- FTS5 は端末の SQLite ビルド依存かつ日本語トークナイザが無いので不使用
- 日本語は**文字bigram**、英数字は単語単位。形態素解析辞書が要らない
- チャンクは500文字・100文字オーバーラップ。Markdown見出しを切れ目として優先

### RAG設定画面（`RagActivity.kt`）
クライアントID入力 / サインイン / 差分同期 / インデックス統計 /
BM25検索テスト。ここだけで動作確認が完結する。

### その他
- `n_ctx` を 1024 → **2048** に戻した。RAGの文脈を積むため。
  8B を使う場合はここが効くので、RAM が足りなければ 1536 に落とす

## セットアップ
`SETUP_ONEDRIVE.md` を参照。Azureアプリ登録が1回だけ必要（無料）。
**「パブリック クライアント フローを許可する」を「はい」にし忘れる**のが
最頻の詰まりどころ。

## 検証の順番
1. RAG設定 → クライアントID貼付 → サインイン
2. 差分同期 → 「文書 N / チャンク M」が増えるのを確認
3. 検索テストでキーワードを入れ、期待する文書が上位に来るか見る
4. OneDrive 側でファイルを1つ編集 → もう一度同期 → **更新 1** だけになるか確認

3 の精度が実用に足りるかが、この設計の分かれ目になる。
足りなければチャンクサイズかトークナイザを調整する。
